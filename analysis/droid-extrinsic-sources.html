DROID Extrinsic Source Analysis

Dataset: DROID (Franka Panda, multi-camera)  |  Date: 2026-04-01

Known Issue: Skeleton alignment quality in DROID rendering varies significantly depending on which camera extrinsic source is ultimately used. Not all cameras in an episode have the same quality — even within the same episode, one camera can be well-aligned while the other is offset.

1. Background: Why Are There Multiple Extrinsic Sources?

To overlay a robot skeleton on a camera image, we need camera extrinsics — the 4×4 transformation matrix that maps 3D points in the robot’s base frame to the camera’s coordinate system.

In DROID, each episode is recorded with 2 external ZED cameras (left_cam and right_cam). However, no single file provides accurate extrinsics for both cameras in all episodes. Instead, the extrinsics come from multiple sources with different levels of accuracy:

1.1 The Three Base Sources

Source File What It Is Accuracy
cam2base cam2base_extrinsics.json Per-episode, per-camera extrinsic from a calibration pipeline. The source field in this file is either "Pred" (model-predicted calibration, ~5.3K episodes) or "GT" (derived from H5 data via IoU matching, ~30.8K episodes). This file only contains 1 camera serial per episode — the other camera needs a different source. Mixed (~60%)
superset cam2base_extrinsic_superset.json A superset file that stores extrinsics for both cameras in ~24K episodes. Each camera has an independent quality metric. This tends to be the most reliable source. Good (~91%)
h5 trajectory.h5 Raw extrinsic stored in the H5 trajectory file. This is the original capture-time calibration, which is often imprecise due to hand-eye calibration drift or lack of refinement. Poor (~33%)

1.2 The Fallback Chain

Our rendering pipeline resolves extrinsics per-camera with a priority fallback chain:

For each camera serial in an episode:

  1. Check cam2base_extrinsics.json — is this serial present?
      YES → use it (source = cam2base)
      NO → continue

  2. Check cam2base_extrinsic_superset.json — is this serial present?
      YES → use it (source = superset)
      NO → continue

  3. Fall back to trajectory.h5
      Use raw H5 extrinsic (source = h5)

Since cam2base only has 1 serial per episode, the typical resolution pattern is:

  • Camera A (found in cam2base) → cam2base
  • Camera B (not in cam2base) → superset or h5

1.3 The Cam2Cam Correction

There is an additional correction step: cam2cam relative transform. When an episode has 2 cameras and a cam2cam calibration file is available:

  • The pipeline picks one camera as the anchor (typically the higher serial number, i.e., right_cam)
  • The other camera’s extrinsic is derived from the anchor using the relative transform between the two cameras
  • This produces a compound source like superset+cam2cam, meaning: “the anchor used superset, and this camera was derived from the anchor via cam2cam”
In theory, cam2cam correction should improve the non-anchor camera by leveraging the anchor’s accurate extrinsic plus a precisely measured inter-camera transform. In practice, the cam2cam transform itself has errors, making this correction unreliable.

1.4 All Six Source Combinations

Combining the 3 base sources with the cam2cam correction, each camera ends up with one of 6 possible sources:

Source Meaning Accuracy (from 40-sample study)
superset Anchor camera using superset extrinsic directly ~91% (10/11 accurate)
cam2base Camera found in cam2base JSON, used directly ~60% (3/5 accurate)
superset+cam2cam Derived from anchor (anchor used superset) ~45% (5/11 accurate)
cam2base+cam2cam Derived from anchor (anchor used cam2base) ~33% (1/3 accurate)
h5 Raw H5 trajectory extrinsic, no patch calibration ~33% (2/6 accurate)
h5+cam2cam Derived from anchor (anchor used h5) 0% (0/4 accurate)

3. Data Distribution

Of the 73,512 total DROID episodes:

Category Episodes Description
Has cam2base entry 36,084 Episode found in cam2base_extrinsics.json (1 serial per episode)
  — source=“GT” 30,790 Extrinsic derived via IoU matching (less precise method)
  — source=“Pred” 5,294 Extrinsic from trained prediction model (generally more precise)
No cam2base entry 37,428 No patch calibration available at all

For the second camera in each episode (not in cam2base):

  • ~20K episodes have superset coverage → second camera uses superset
  • ~10K episodes have no superset → second camera falls back to h5
Key insight: The source field in cam2base_extrinsics.json ("GT" vs "Pred") is not a reliable predictor of rendering quality. Both GT and Pred episodes can produce good or bad results. What matters is the actual extrinsic source used per-camera after the fallback chain resolves.

4. Conclusions & Recommendations

Conclusion: The previous episode filter (source == "GT") was filtering based on the wrong dimension. It discarded ~30K episodes that could include many usable cameras.

Recommended new filtering strategy:

Extrinsic Source Action Rationale
superset (direct) Keep ~91% accuracy, most reliable source
cam2base (direct) Keep with caution ~60% accuracy, may need per-episode quality threshold
superset+cam2cam Evaluate further ~45% accuracy, cam2cam correction is unreliable
h5, h5+cam2cam, cam2base+cam2cam Discard ≤33% accuracy, not worth the noise

This approach would shift from filtering ~85% of episodes (the old GT filter) to a per-camera source check, potentially unlocking 20K+ additional usable camera-episode pairs while maintaining quality.
