Skeleton Near-Clip Fix

Affected: DROID (close-up cameras)  |  Date: 2026-03-25

Issue: When the camera is very close to the robot, skeleton segments near the camera were not rendered, causing the lower portion of the skeleton to disappear.

1. Problem

The skeleton rendering code applies a near clipping plane to avoid visual artifacts from perspective projection. Any bone segment with an endpoint closer than the near distance is skipped. The original near clip was set to 0.05m (5cm).

In DROID, the external camera 22008760 is mounted very close to the Franka robot. At many frames, the majority of the kinematic chain falls within 5cm of the camera or behind it entirely:

DROID camera 22008760 — Joint Z-distances in camera frame (frame 0):

  panda_hand:  z_cam =  0.011m   (in front, but < 0.05m near clip)
  panda_link8: z_cam =  0.011m   (in front, but < 0.05m near clip)
  panda_link7: z_cam = -0.035m   (behind camera)
  panda_link6: z_cam = -0.049m   (behind camera)
  panda_link5: z_cam = -0.049m   (behind camera)
  panda_link4: z_cam = -0.174m   (behind camera)
  panda_link3: z_cam = -0.158m   (behind camera)
  panda_link2: z_cam =  0.015m   (in front, but < 0.05m near clip)
  panda_link1: z_cam =  0.015m   (in front, but < 0.05m near clip)
  panda_link0: z_cam =  0.170m   (in front, OK)

With near = 0.05m, only panda_link0 is reliably in front of the near plane. The skeleton appears as just the base link, with the rest of the arm missing.

2. Why Near-Clip Exists

Near clipping is necessary for the gripper rendering (coordinate axes + finger lines). The gripper draws long lines from the EE origin to axis endpoints (0.2m away). When the origin is behind the camera, the perspective projection flips, and these long lines cross the entire screen:

Gripper rendering (axis lines):
  origin ────── 0.2m ──────► X-axis tip
  origin ────── 0.2m ──────► Y-axis tip
  origin ────── 0.2m ──────► Z-axis tip

If origin is behind camera:
  - Projected origin is at flipped, extreme coordinates
  - Lines from flipped origin to visible tips cross the entire image
  - Visual artifact: colored lines spanning the full frame

Skeleton bone segments are much shorter (joint-to-joint, typically 5-15cm), so the same artifact is less severe.

3. Fix: Separate Near-Clip for Skeleton vs Gripper

Solution: Use different near-clip thresholds:
  • Skeleton: near = 0.001m (1mm) — allows drawing bones very close to camera
  • Gripper: near = 0.05m (5cm) — prevents axis line explosion

Additionally, for segments that cross the near plane (one endpoint in front, one behind), the segment is clipped to the near plane via linear interpolation, rather than being discarded entirely:

Before (discard entire segment):
  if p0.z < near OR p1.z < near:
      skip  # Both points or either point too close → skip

After (clip to near plane):
  if p0.z < near AND p1.z < near:
      skip  # Both behind → truly invisible
  elif p0.z < near OR p1.z < near:
      # One in front, one behind → interpolate to near plane
      t = (near - p0.z) / (p1.z - p0.z)
      clipped_point = p0 + t * (p1 - p0)
      draw from clipped_point to the visible endpoint

Implementation in dataset/common/render_utils.py, function render_skeleton_frame():

def _clip_to_near(pa, pb):
    a_ok = pa[2] >= near
    b_ok = pb[2] >= near
    if a_ok and b_ok:
        return pa, pb
    if not a_ok and not b_ok:
        return None
    t = (near - pa[2]) / (pb[2] - pa[2])
    clipped = pa + t * (pb - pa)
    return (clipped, pb) if not a_ok else (pa, clipped)

5. Remaining Limitations

Bone segments where both endpoints are behind the camera (Z < 0) still cannot be rendered — this is a fundamental geometry limitation, not a bug. For DROID camera 22008760, links 3-7 are frequently all behind the camera simultaneously. These segments are invisible from this viewpoint regardless of the near-clip threshold.

For datasets where close-up cameras are the primary view, the recommendation is to prefer farther cameras (e.g. camera 24400334 for DROID) or accept partial skeleton visibility on close-up views.
