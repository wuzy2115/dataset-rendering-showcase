4. Root Cause Analysis

4.1 FK Verification

FK-computed TCP vs dataset TCP shows a constant 71mm offset (tool length) across all frames — confirming joint angles and URDF kinematics are correct:

t=0.0: FK_tcp=[+0.628,-0.114,+0.303] DS_tcp=[+0.631,-0.115,+0.232] diff=71mm
t=0.1: FK_tcp=[+0.515,-0.013,+0.171] DS_tcp=[+0.519,-0.013,+0.101] diff=71mm
t=0.5: FK_tcp=[+0.705,+0.010,+0.123] DS_tcp=[+0.718,+0.006,+0.054] diff=71mm
t=0.9: FK_tcp=[+0.542,-0.133,+0.211] DS_tcp=[+0.541,-0.132,+0.140] diff=71mm

4.2 Camera Extrinsic Chain

OptiTrack measures:
  - Camera position + quaternion in OptiTrack world frame
  - NIST_Board1 position + quaternion (≈ robot base area)

Current pipeline:
  T_base_in_optitrack = T_nist_board    (assumption)
  T_cam_in_base = inv(T_base) @ T_cam
  extrinsic = inv(T_cam_in_base)

Problem:
  NIST_Board1 position [0.379, 0.201, 0.006] is NOT exactly the robot base.
  The board is placed on the table near the robot but offset by an unknown amount.

4.3 Missing Parameters

NIST → Robot base offset Causes uniform translation of entire skeleton Not documented. Could be solved by manual pixel annotation + PnP, or by contacting dataset authors.
Session-specific pose files Episode 2025-01-10 has no matching pose file (only 01-09 sessions available) Full dataset on Google Drive may include all sessions.

5. Conclusion

Status: Not fully renderable due to missing camera intrinsics and unknown robot-base-to-NIST-board offset. The skeleton kinematics (FK) are verified correct. With the correct base position, rendering would work immediately.

To make this dataset fully renderable:

  1. Obtain camera intrinsics (calibrate HAMA cameras, or request from authors)
  2. Determine NIST board → robot base offset (measure physically, or solve via PnP with manual pixel annotations)
  3. Download all session pose files from the full dataset release
