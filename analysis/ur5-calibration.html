UR5 Calibration Analysis

Dataset: RH20T cfg3 (UR5 + Robotiq 2F-85)  |  Date: 2026-03-25

Known Issues: Skeleton alignment varies by camera due to per-camera extrinsic calibration quality. Gripper orientation differs from actual due to UR5 URDF ee_link frame convention mismatch.

1. Summary

RH20T UR5 (cfg3) episodes exhibit inconsistent skeleton-to-image alignment when rendered using base-frame Forward Kinematics (FK). Some episodes show correctly aligned skeletons; others show a clear offset. Through systematic investigation, we determined this is caused by per-camera extrinsic calibration quality variation, not by code bugs or joint angle errors.

Additionally, the gripper orientation rendered from FK ee_link does not match the actual gripper direction, because the UR5 URDF ee_link coordinate frame has a different orientation than the dataset's TCP coordinate frame.

2. Rendering Pipeline

The current rendering pipeline for RH20T (and all fixed-base robots) uses base-frame FK:

Joint Angles (from dataset)
    │
    │  Forward Kinematics (URDF)
    ▼
3D positions of each link in URDF base frame
    │
    │  Camera Extrinsic (world-to-camera, from RH20T calibration)
    │  Camera Intrinsic
    ▼
2D pixel coordinates on image

This pipeline has two key assumptions:

  1. The URDF base_link frame = the "world/base" frame that the extrinsic maps from
  2. The extrinsic calibration accurately represents this mapping
If either assumption fails, the projected skeleton will appear offset from the actual robot in the image.

5. Gripper Orientation Issue

5.1 UR5 URDF ee_link vs TCP Orientation Mismatch

The gripper rendering uses FK ee_link's rotation to determine the gripper finger direction. However, the UR5 URDF's ee_link coordinate frame has a significantly different orientation than the dataset's TCP frame:

Dataset TCP rotation matrix:

[[-1.000, -0.008, -0.000],
 [-0.008,  1.000,  0.009],
 [ 0.000,  0.009, -1.000]]

FK ee_link rotation matrix:

[[ 0.002, -0.005,  1.000],
 [ 0.021,  1.000,  0.005],
 [-1.000,  0.021,  0.002]]

These are clearly different coordinate frame conventions. The gripper rendering assumes fingers open in the X-Z plane with Z as the forward direction, which matches the TCP frame but not the FK ee_link frame.

5.2 Validation: Episode Without Skeleton

Episode 08 (task_0005_user_0016_scene_0009_cfg_0003) had no joint angles available, so the gripper was rendered using the raw TCP pose (fallback). The gripper orientation was correct in this case, confirming that the TCP frame convention matches the gripper rendering assumptions, while the FK ee_link frame does not.

Episode 08 (TCP fallback, no skeleton) shows correct gripper orientation — visible in the reference videos below.

7. Conclusion & Proposed Solutions

ProblemRoot CauseScope
Skeleton offset (some episodes) Per-camera extrinsic calibration quality varies. Camera 104122063550 has inaccurate base-aligned extrinsics. UR5 cfg3 only (Franka cfg5 and KUKA cfg7 cameras have better calibration)
Gripper orientation wrong UR5 URDF ee_link coordinate frame orientation differs from dataset TCP frame orientation UR5 cfg3 only

Proposed Solutions

Fix 1 — Skeleton offset: Use visibility filtering (require >50% of skeleton joints visible in frame) to automatically exclude episodes rendered with poorly-calibrated cameras. Alternatively, use EE-relative FK anchored on the dataset's TCP pose (accepting a ~25cm tool offset).
Fix 2 — Gripper orientation: Use the TCP pose's rotation (from dataset) for the gripper rendering instead of the FK ee_link rotation. Position the gripper at the FK ee_link location (to connect with skeleton) but apply TCP rotation for the finger direction.
