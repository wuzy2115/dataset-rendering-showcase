  1. Data Adaptation — Convert dataset-specific formats (quaternion convention, extrinsic direction) to standardized inputs
  2. URDF Resolution — Look up robot model in registry.json, get URDF path + kinematic chain spec
  3. Skeleton Rendering — render_skeleton_frame(): FK through URDF chain using joint angles, project to image via camera calibration
  4. Gripper Rendering — render_gripper_frame(): Draw coordinate axes + gripper fingers at FK end-link pose (not raw TCP, to avoid gap with skeleton)
  5. Video Encoding — PyAV with H.264, not cv2.VideoWriter (which fails silently in many environments)

Key requirement for rendering: Camera intrinsics + extrinsics (for 3D→2D projection), joint angles (for skeleton FK), and URDF in registry. Missing any of these degrades to gripper-only or top-down trajectory plot.

DataLoader — Embodiment-Based Multi-Dataset Sampling

When training on multiple datasets jointly, frame counts vary by orders of magnitude (from ~250K to ~12M frames). Sampling proportional to frame count lets the largest datasets dominate and leaves small datasets severely undertrained. This section describes the temperature sampling strategy we use and its effect on per-dataset sampling probabilities.

1. Dataset Overview

The training mix consists of 6 dataset configurations spanning 3 robot embodiments, totaling approximately 28M frames.

2. Temperature Sampling

A temperature parameter T controls how uniform the sampling distribution is across datasets:

  • Weight formula: weight_i = total_frames_i ^ (1/T), then normalized to sampling probabilities
  • T = 1: proportional to frame count — large datasets dominate
  • T = 2: sqrt weighting — moderate balance
  • T = 3: current default — empirically chosen balance point
  • T → ∞: fully uniform — each dataset sampled equally

Sampling uses a two-level hierarchy: first sample an embodiment by weight, then sample a dataset within that embodiment by weight. The embodiment weight equals the sum of its constituent dataset weights, making this exactly equivalent to flat temperature sampling across all datasets.

Benchmark Construction 1,000 Episodes

We construct a held-out evaluation benchmark of 1,000 episodes (284,037 frames) across 6 datasets for the world model evaluation. The benchmark is drawn exclusively from quality-filtered, deduplicated episodes (manifest.json["kept"]) — dropped episodes (moving camera, near-duplicates) are never used, as near-duplicate leakage would bias metrics upward. The split is a separate offline step that runs after the full data processing pipeline (camera filter → dedup → render → caption → episode_meta) and does not modify the pipeline's manifest.json.

Design Context

Multiple baselines will run inference over this benchmark, so total size must be small enough for practical inference (~1,000 samples). Metrics are reported per dataset (not merged by embodiment), because datasets differ in origin (real vs. sim), scene background, and collection setup.

Related work precedents: Genie Envisioner (EWMBench) uses 10 tasks × 100 samples = 1,000 total with tasks fully disjoint from pretraining; Kinema4D uses a stratified ~2% hold-out (3,200 / 201K) with per-source reporting; DreamZero uses task-level seen/unseen split with per-task reporting. We rejected task-level held-out splits because OOD tasks with unseen action renderings would produce systematically poor results, confounding model quality with distribution shift.

Construction Pipeline

1
Budget Allocation (√N weighting)
Each dataset receives a share of the 1,000-episode budget proportional to √N, where N is the number of kept episodes after quality filtering. This prevents large datasets (agibot: 94K episodes) from dominating while still giving them more representation than tiny ones (rh20t_cfg7: 557 episodes). The √N weighting is consistent with the DataLoader's embodiment-level sampling philosophy. Per-dataset counts can be overridden via config YAML.
2
Feature Extraction (caption + motion)
For each episode we extract two signals:
Semantic: VLM-generated captions (from caption.pickle) are embedded using all-MiniLM-L6-v2 (384-dim), then reduced to 2D via PCA. Episodes with unreadable or missing captions are skipped entirely.
Motion: End-effector trajectory path length from episode_meta.npz — sum of frame-to-frame Euclidean displacements of the EE translation vector. For dual-arm robots (agibot), left and right arm trajectories (ee_pose_left / ee_pose_right) are summed. Missing motion data is imputed with the dataset median (episode kept if caption is valid).
The 3D feature vector (2D semantic + 1D motion) is projected to 2D via a second PCA, then min-max normalized to [0, 1].
3
Coverage-Based Diverse Subset Selection
We need to select K episodes (= allocated budget) that are maximally spread out in the 2D feature space. To do this, we partition the space into K regions using K-means (where K is the desired sample count, not the number of "natural clusters") and pick the episode closest to each region's centroid (medoid). This is a standard coverage sampling technique — K-means is used as a space-partitioning tool, not for cluster discovery. The result: episodes are as diverse as possible across both task semantics and motion complexity, so rare patterns (bi-manual coordination, tool use, long-horizon tasks) get equal representation alongside common ones (e.g., "pick up object from table").
4
Three-Manifest Split
Each dataset directory ends up with three manifests, all using the same "kept" list format that EmbodimentDataLoader already reads — zero DataLoader code changes required:
manifest.json — pipeline output (never modified)
benchmark_manifest.json — selected benchmark episodes
train_manifest.json — kept − benchmark = training episodes
The split is deterministic (seed=42) and fully reproducible. Benchmark episodes are exported via hardlinks to a dedicated directory with per-episode files: rgb.mp4, overlay.mp4, gripper_scenario.mp4, caption.pickle, episode_meta.npz.

Method 1: Ours — Cosmos2 Concat

The simplest approach. Action condition (an RGB skeleton/overlay video) is encoded through the same VAE as the target video, producing a 16-channel latent. This is concatenated to the noisy video latent along the channel dimension, then passed through a separate PatchEmbed (Rearrange + Linear, no nonlinearity) and added to the main embedding with a scaling factor of α = 0.1.

// Data flow
condition_video (B,3,T,H,W) → VAE encode → control_latent (B,16,T',H',W')

// In forward():
x = cat([noisy_latent(16ch), mask(1ch), control_latent(16ch)], dim=1) // 33ch total

// In prepare_embedded_sequence():
main_embed  = x_embedder(x[:, :17])       // Rearrange+Linear: 17ch → 2048d
ctrl_embed  = additional_x_embedder(x[:, 17:]) // Rearrange+Linear: 16ch → 2048d
tokens = main_embed + 0.1 * ctrl_embed   // α-weighted addition

// Then tokens go through 28 DiT blocks (no further action injection)
for block in dit.blocks: tokens = block(tokens, text_kv, timestep)

Method 2: LOME — Wan2.1 VACE + Camera Control

Uses VACE (All-in-One Video Creation and Editing), Alibaba's official conditional control framework pretrained on multi-task video editing (inpainting, outpainting, reference-to-video, etc.). LOME repurposes it by feeding 2D hand skeleton renderings as VACE input. VACE is a separate 8-layer transformer (same block structure as DiT) that runs independently, producing 8 hint features injected into the main DiT at blocks 0, 5, 10, 15, 20, 25, 30, 35 via residual addition. Additionally, camera control uses Plucker ray embeddings processed through a SimpleAdapter (PixelUnshuffle + Conv2d + ResBlock), added once at the patch embedding stage.

Method 3: Kinema4D — Wan2.1 Dual-Path Cross-Attention

The most complex approach with two injection paths. Path 1: arm features are directly mixed into the hidden state channels before patchify (zero-init Linear for features, direct replacement for mask). Path 2: arm condition is separately embedded via Conv3d(20→128, k=1×15×15), projected to 5120d, and concatenated with CLIP image tokens to serve as cross-attention KV in all 40 DiT blocks. Notably, Kinema4D replaces text conditioning with arm tokens in the cross-attention — the original text KV projections (to_k/to_v) are reused for arm, while image uses dedicated add_k_proj/add_v_proj.

Key Design Insights

1
Injection Depth vs. Compute Tradeoff
Ours injects once (cheapest), LOME injects at 8 layers via a parallel network (~20% overhead), Kinema4D injects at every layer via cross-attention (moderate overhead since it reuses existing KV projections). Deeper injection generally gives stronger controllability but costs more compute.
2
VACE: Pretrained Condition Encoder
LOME's VACE is pretrained on large-scale multi-task video editing (Alibaba's official release, ICCV 2025), giving it strong priors for interpreting visual conditions. The 8 VACE blocks are full DiTBlocks with independent (non-shared) weights, producing skip hints at each layer. For 14B, VACE uses 8 blocks at intervals of 5 DiT layers — the VACE paper's ablation showed distributed injection outperforms dense injection, and larger models need sparser conditioning.
3
Zero-Init: Safe Fine-Tuning
Kinema4D zero-initializes both arm_cond_proj and arm_patch_proj, so at training start the model behaves identically to the pretrained checkpoint. Condition influence grows gradually as weights update. This is critical when injecting into a pretrained model — random-init projections would corrupt the pretrained features immediately.
4
Plucker Embedding for Camera
LOME is the only method with explicit camera conditioning. Plucker coordinates encode both ray direction and position in a 6D vector per pixel. The 6ch embedding is temporally packed (4 frames → 24ch) to match VAE's temporal compression, then processed by a SimpleAdapter and added at patchify. Kinema4D uses pointmaps (concatenated along width) as an implicit geometric representation instead.
